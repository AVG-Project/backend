СОЗДАНИЕ пользователя через апи:
...api/v1/auth/users/   #При наличии авторизации выдает список из 1 пользователя который запросил, если пользователь имеет статус сотрудника выдает всех из бд.(всегда список)
...api/v1/auth/users/ <id:int>/  #При добавлении айди GET выдает только 1 юзера. В этом ендпоинте доступно PUT и PATCH. 
...api/v1/auth/users/me/  #Выдает данные 1 авторизированного пользователя

постим данный в формате:


POST  ...api/v1/auth/users/ 
{
    "email": "eminence_grise@inbox.ru",  # для теста лучше создай настоящий эмеил, будут письма автоматом приходить.
    "phone": "+70000000012",  # сделал поле необязательным и может быть пустым. 
    "password": "123123xx",
    "re_password": "123123xx",  # проверку лучше сделать на фронту но поле все равно требует
    "last_name": "eminence_grise",
    "first_name": "eminence_grise",
    "patronymic": "eminence_grise",
    "birth_date": "2001-01-01",  # данный формат точно принимает
    "mailing": true,
    "personal_data_processing": true,
    "registration_by_code": ""  # если регистрация была через ссылку лояльности. Забираешь от туда код и вставляешь сюда.
}

PS. Я вроде все ошибки потестил, ну там тот же email и телефон. Ошибки там выдаются по полям к которым ошибка относится. Потести и прикрути. Наверно если статус 400 забираешь ошибку из поля в котором она допущена и к нему прикручиваешь. 

АКТИВАЦИЯ
После создания юзера автоматически отправляется письмо для активации аккаунта. До этого за него пока нельзя авторизироваться. Приходит такая ссылка:
<ИМЯ ДОМЕНА>/auth/verify/MjM/ceon9m-bb4d52a4c4a605c6c79aacbc5e6ffcba/
После используем апи:
POST на .../api/v1/auth/users/activation/  # апи активации акку
{
    "uid": "MjM",  # из ссылки
    "token": "ceon9m-bb4d52a4c4a605c6c79aacbc5e6ffcba"   # из ссылки
}
После ответа 204 аккаунт становится активным. Статус активности позволяет уже авторизироваться. После этого шага кидай на вход. 

Если нужно отправить письмо активации еще раз:
POST .../api/v1/auth/users/resend_activation/  #
{
    "email": "eminence_grise@inbox.ru"
}
Придётся держать введенный емеил в памяти или кеше. Моя система позволяет отслеживать сессию неавторизованного юзера. Но кроме ключа сессии ничего не хранит и то удаляет.



РЕДАКТИРОВАНИЕ
Для редактирования полей не используемых в авторизации используй только PATCH. 

PATCH ...api/v1/auth/users/ <id:int>/  # редактирование всех полей кроме password и email.(не будет письма активации)
{
    "phone": "+70000000012",  # сделал поле необязательным и может быть пустым. 
    "last_name": "eminence_grise",
    "first_name": "eminence_grise",
    "patronymic": "eminence_grise",
    "birth_date": "2001-01-01",
    "mailing": true,
    "personal_data_processing": true,
}

РЕДАКТИРОВАНИЕ ПОЧТЫ
PUT ...api/v1/auth/users/ <id:int>/  # редактирование email и phone через этот метод.
Этот метод снимет активацию с аккаунта и отправит письмо с активацией. Так что после редактирования email пусть возникает окно как при регистрации, что нужно пройти по ссылке. А там порядок описанный выше. После этот метода в аккаунт нельзя будет войти, пока он не будет активирован.
{
    "email": "eminence_grise@inbox.ru",
    "phone": "+70000000012",  
    "last_name": "eminence_grise",   # тоже нужно
    "first_name": "eminence_grise",  # тоже нужно
    "birth_date": "2001-01-01"  # тоже нужно
}

РЕДАКТИРОВАНИЕ ПАРОЛЯ (если юзер авторизирован)

POST  .../api/v1/auth/users/set_password/

{
    "new_password": "123123123xx",
    "re_new_password": "123123123xx",
    "current_password": "123123xx"
}

ВОССТАНОВЛЕНИЕ ПАРОЛЯ (без авторизации)

POST .../api/v1/auth/users/reset_password/
{
    "email": "eminence_grise@inbox.ru"
}
На почту приходит такая ссылка:
http://127.0.0.1:8000/password/reset/confirm/MjM/ceoqey-fda5179252b0110df0418a66e210e123
После перехода по ней запрашиваешь апи:
POST  .../api/v1/auth/users/reset_password_confirm/
{
    "uid": "MjM",  # из ссылки
    "token": "ceoqey-fda5179252b0110df0418a66e210e123",  # из ссылки
    "new_password": "123123xx",
    "re_new_password": "123123xx"
}
